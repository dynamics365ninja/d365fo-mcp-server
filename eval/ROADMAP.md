## Fix the grounded modify surface (blocks honest scoring)

The 2026-07-21 capture sweep captured 9 goldens and lifted core coverage
79.1% → 97.7%, but it also established that **the grounded tool path can no
longer complete a composite case**. Full evidence, 34 numbered defects:
`docs/eval-sweep-findings-2026-07-21.md`. Progress 2026-07-22:

- **Root cause confirmed (finding #29 cluster):** bridge-backed `modify` ops
  resolve the target BY NAME against a **startup-fixed** DiskProvider model
  (`MetadataWriteService.cs` `_provider.Tables.Read(name)`), so an object
  written THIS session isn't in the model and fails `could not resolve`.
  `filePath` never reaches the bridge — it only steers the TS-side file lookup
  (proves #23). **The real C# fix — reload same-session on-disk objects into
  the provider — CANNOT be validated in-repo:** the bridge won't compile/run
  without the VM's `Microsoft.Dynamics.AX.Metadata` assemblies. It is a
  **VM-side task**, not a repo-side improver fix.
- **Partial repo-side closure landed:**
  - `add-control` design-root addressing — **PR #728**
    (`fix/form-design-root-add-control`), also fixes form `create` #9/#10 and
    the #11 misleading-diagnostic (no longer recommends `create overwrite=true`
    as a modify remedy).
  - `add-index` same-session — **PR #729**
    (`fix/modify-same-session-add-index`): TS direct-XML fallback
    `directXmlAddIndex` mirrors the existing fallbacks, so a same-session table
    index no longer forces whole-file overwrite.
  - ⚠️ **#729 is currently ORPHANED FROM `main`.** It was merged (03:04) into
    its base branch `fix/form-design-root-add-control` — but #728 had already
    merged to `main` separately (02:49), so #729's commit (`e94835f`) never
    reached `main`. `origin/main` has ONLY #728. To land the add-index fix,
    re-merge #729 onto `main` (cherry-pick `e94835f` or reopen a PR based on
    `main`). Not needed for the 3 form cases, but required before any
    add-index-based capture.
  - **The form fix requires `git pull` FIRST:** the running bridge was rebuilt
    from local `main` (still `c262b19`, behind `origin/main` by 2), so it does
    NOT contain #728 — the 2026-07-22 form probe confirmed `add-control` still
    can't address the design root. Pull `origin/main` → rebuild the bridge →
    restart the MCP server, then the 3 form cases can run.
- **Still open (deferred on purpose):** `add-field-group` / `modify-field`
  (same #29 family) have NO direct-XML fallback yet — their canonical element
  ordering can't be verified VM-free and getting it wrong silently corrupts
  (finding #13). Until the VM-side bridge resolution fix lands, #728's honest
  diagnostic is the safer interim; table cases needing those two ops still hit
  overwrite.
- **Two committed goldens enshrine defective output**, so those cases can
  never fail on what they exist to catch: `L4-entity-security`
  (duty/role emit `AxSecurityRolePermissionSet` instead of
  `AxSecurityPrivilegeReference` — xppbp proves the chain is dead, #31) and
  `L1-form-detailsmaster` (missing `<DataSource>` sibling; its own corpus
  record already has `build.succeeded=false`, #32). Both need re-capture after
  the writers are fixed.

Related measurement trap: only `fullBuild` runs metadata validation — an
incremental build reported 0 errors on a model with 2 real metadata errors, so
`pass@build` on incremental runs is weaker than it looks.

Scorer-integrity fix landed 2026-07-22: finding #1 (a genuinely missing

## Coverage closure loop (VM, ongoing, ~2–4 leaves/week)

The machinery already exists; this just gives it a queue ordered by
matrix-gap × frequency weight: `eval-author` drafts a case for each leaf
missing E → `eval-run` captures the golden on the VM → MODEL_ERROR clusters
flow through `knowledgeFeedback` into proposed knowledge entries (human
review stays mandatory, as the module enforces) → TOOL_DEFECT/VALIDATOR_GAP
go down the standard improver path. For leaves where model training data is
weak, knowledge entries get a canonical minimal example mined from the real
AOT (example mining), not written from memory.
